# DOCUMENT 2 - SYSTEM ARCHITECTURE SPECIFICATION

# PoolPredict

## Architecture Principles

### 1. Shared Tournament Data

Tournament data is global and shared across the platform.

Examples:

* FIFA World Cup 2026
* UEFA Euro 2028
* Champions League

Fixtures, scores and results are imported once and reused by all pools.

Benefits:

* Minimal external API usage
* Consistent event data
* Simplified synchronization

### 2. Pool as Tenant Boundary

Pools are the primary tenant unit.

Everything below is pool-specific:

* Members
* Settings
* Markets
* Predictions
* Leaderboards
* AI Reports

### 3. Pool Owners Are Not Bookmakers

Pool owners should not:

* Create fixtures
* Enter scores
* Update results
* Adjust lines manually for every match

Pool creation should be mostly automated.

### 4. Local Database Is Source Of Truth

External providers are data sources only.

Application reads from MariaDB.

Frontend never reads external APIs directly.

# Database Design

## IDs

Use UUID v7 for primary keys.

Store UUIDs as `binary(16)` where the ORM and database tooling support it.

Otherwise use:

```sql
char(36)
```

Primary keys must be UUID-based.

Avoid auto-increment IDs.

Rationale:

* UUID v7 is time-ordered and works better for indexed inserts than random UUID v4
* UUIDs are safer to expose through APIs and URLs than sequential numeric IDs
* IDs can be generated by background jobs without relying on one database sequence
* A consistent ID strategy keeps cross-module references simpler

High-volume append-only tables, such as `point_ledger`, `settlement_logs` and audit logs, still use UUID v7 primary keys.

For those tables, the primary key is mainly a stable row identity. Most reads should use access-pattern indexes instead of scanning by ID.

Required indexes should be based on query patterns, for example:

* `pool_id`
* `member_id`
* `prediction_id`
* `settlement_run_id`
* `event_id`
* `created_at`
* Composite indexes such as `(member_id, created_at)` and `(pool_id, created_at)`

If profiling later proves UUID primary keys are a real bottleneck for append-heavy tables, an internal `bigint` identity column may be added as a storage optimization while keeping UUIDs as the public and cross-module identifiers.

# Settlement Engine

Requirements:

* Pure deterministic calculations
* No AI
* No external calls
* Fully unit tested
* Required before MVP release so submitted predictions can resolve into final point balances
* Idempotent settlement so reruns cannot double-credit or double-debit members
* Append-only point ledger entries for payouts, refunds and corrections
* Use prediction snapshots for line values and payout configuration

Supported Results:

* Win
* Lose
* Push
* Half Win
* Half Lose
* Cancelled

Supported Markets:

* Winner
* Handicap
* Over/Under
* Odd/Even
* Correct Score

Point Rules:

* Prediction stake is deducted immediately when a prediction is submitted
* Negative balances are allowed
* Members with negative balances cannot submit new predictions
* Push and cancelled outcomes refund the submitted stake
* Winning outcomes credit stake plus configured payout
* Losing outcomes keep the submitted stake deducted
* Payout uses the configuration snapshotted when the prediction was submitted

Market Lifecycle:

* Draft
* Open
* Locked
* Settled
* Voided

Event Lifecycle:

* Scheduled
* Live
* Finished
* Postponed
* Cancelled
* Settled

# Point Payout Configuration

PoolPredict does not use public betting odds.

External sports providers provide fixtures, participants, kickoff times, scores and statuses only. They do not provide market payout values for MVP.

Platform Admin manages global default point payout configuration.

Global defaults define:

* Market types enabled by profile
* Default line values
* Supported line sets
* Payout multipliers
* Push, half-win, half-lose and cancellation behavior

When a new match is created, markets are generated from the current global defaults.

Before a market is locked, Platform Admin can change line values for that match's markets.

Pool owners cannot manually manage line values or payout configuration.

Submitted predictions must snapshot:

* Market type
* Selected option
* Line value
* Payout multiplier
* Stake
* Payout configuration version

Settlement must use the snapshotted prediction values instead of the current market configuration. This prevents later admin changes from changing already-submitted predictions.

Future extension:

* Tournament-specific payout configuration can override global defaults
* Pool-specific payout configuration is not part of MVP

Suggested tables:

* payout_configurations
* payout_configuration_market_rules
* market_line_overrides
* prediction_snapshots

# Localization

English is the default language and the initial production language.

User-facing labels, market names, system messages, recap labels and admin action names must use translation keys instead of hard-coded display text.

Platform admins can add languages later by entering translations for each English source term.

Missing translations must fall back to English.

Suggested tables:

* languages
* translation_terms
* translation_values
